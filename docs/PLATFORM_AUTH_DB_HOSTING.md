# Платформа: авторизация, БД, хостинг, оплата, админка

Цель документа — зафиксировать **что реально сделать на бесплатных тарифах**, что **стоит денег**, и **в каком порядке** внедрять, не ломая текущий web MVP (`web/`, данные в `localStorage`).

---

## 1. «Бесплатно» — что имеется в виду

| Компонент | Бесплатные варианты | Оговорки |
|-----------|---------------------|----------|
| Хостинг фронта (SPA) | Vercel, Netlify, Cloudflare Pages | Лимиты трафика/сборок, без «бесконечного» SLA |
| Бэкенд + API | Supabase, Neon + serverless functions, Railway/Render free tier | Сон/лимиты CPU на free, cold start |
| PostgreSQL | Supabase, Neon, Railway | Квоты размера БД и соединений |
| Авторизация (email/OAuth) | Supabase Auth, Clerk free tier, Firebase Auth | Лимиты MAU |
| **2FA по SMS** | Практически нет «бесплатно навсегда» | Twilio/MessageBird и аналоги берут за SMS; на тестах — копейки, в проде — постоянный расход |
| **2FA без SMS** | TOTP (Google Authenticator, и т.п.) | Обычно **без пересылки SMS**, зато бесплатно для пользователя и дешевле для вас |
| Платежи (подписка) | Stripe / ЮKassa и т.д. | Регистрация бесплатна; **комиссия с каждой оплаты**; тестовый режим бесплатен |
| Админка | То же приложение + роль `admin` в БД | Разработка — время; хостинг — как у основного сайта |

**Вывод:** «Всё бесплатно» возможно как **старт на free tier’ах** и **TOTP вместо SMS** на первом этапе. SMS-2FA в проде почти всегда **платная** интеграция + иногда требования к персональным данным.

## 2. Рекомендуемый стек (минимум сервера своими руками)

Один популярный путь для соло/малой команды:

1. **Supabase** (или аналог): PostgreSQL + Auth (email magic link / OAuth) + Row Level Security + опционально Edge Functions для webhooks оплаты.
2. **Фронт** (текущий Vite React): деплой на **Vercel/Netlify**, переменные окружения с URL Supabase.
3. **Оплата:** **Stripe Billing** (международно) или **ЮKassa** (РФ) — webhook → обновление `subscriptions` в БД.
4. **Админка:** маршрут `/admin` (или отдельный поддомен), доступ только если `profiles.role = 'admin'`, страницы: пользователи, подписки, агрегаты (SQL views / materialized views), логи событий в таблице `audit_log`.

Альтернатива: свой **Node (Fastify/Nest) + Prisma + PostgreSQL** на Railway/Render — больше контроля, больше кода.

## 3. Схема данных (черновик)

Условные таблицы (уточняются под продукт):

- `profiles` — `id` (uuid, = auth user), `email`, `phone` (опционально), `role` (`user` | `admin`), `plan` (`free` | `premium`), `created_at`
- `transactions` — привязка к `user_id`, поля как в текущей модели
- `budgets` — `user_id`, категория, лимит, период
- `subscriptions` — `user_id`, `provider` (`stripe` | `yookassa`), `status`, `current_period_end`, `price_id`
- `audit_log` — `admin_id`, `action`, `entity`, `metadata`, `created_at` (для «красивых логов» в админке)

Миграция с **localStorage** → облако: однократный импорт после логина или синхронизация «пустой облачный аккаунт + импорт JSON».

---

## 4. Безопасный протокол

- Всё общение клиент ↔ Supabase/API — **HTTPS**.
- Сессии: JWT от провайдера или cookie с httpOnly (если свой бэкенд).
- **2FA:**  
  - **Этап 1:** TOTP (бесплатно для продукта).  
  - **Этап 2:** SMS через провайдера с оплатой за сообщение + политика ПДн.

---

## 5. Страница оплаты и описание Premium

Отдельный маршрут `/pricing` или `/subscribe`:

- Таблица сравнения **Free vs Premium** (из `docs/MONETIZATION.md` + уточнение фич).
- Кнопка «Оформить» → Stripe Checkout / платёжная форма провайдера → webhook → запись в `subscriptions`.

Тексты и список фич лучше держать в i18n, как остальной UI.

---

## 6. Админ-панель

Минимум:

- Список пользователей (поиск, фильтр по плану).
- Подписки и статусы.
- Агрегаты: регистрации за день/неделю, MRR (если есть данные от оплат).
- Лента `audit_log` + фильтры.

«Красивые логи» = таблица в БД + простая визуализация (таблица + графики позже через Metabase/Grafana или встроенные чарты).

---

## 7. Деплой «доступно всем»

1. Репозиторий в GitHub/GitLab.
2. Подключить Vercel к репо, root = `web/`, `npm run build`, output `dist`.
3. Включить env: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (или ваш API URL).
4. Кастомный домен — в настройках хостинга (часто бесплатно на первом уровне).

## 8. Реалистичные фазы (что делать по порядку)

| Фаза | Содержание |
|------|------------|
| **A** | Проект Supabase (или Neon + Auth), схема таблиц, RLS, регистрация/вход с фронта |
| **B** | Синхронизация транзакций/бюджетов с БД вместо только localStorage |
| **C** | TOTP 2FA |
| **D** | Stripe/ЮKassa + страница `/pricing` + webhook → `subscriptions` |
| **E** | Админка + `audit_log` |
| **F** | SMS 2FA (опционально, когда есть бюджет) |

Сделать **всё из списка за один заход «бесплатно»** нереалистично по времени и рискам; **фазы A–B** уже дают облачного пользователя и основу под монетизацию.

---

## 9. Ответ на вопрос «сможем ли сейчас всё это бесплатно?»

- **С технической стороны:** да, **начать** можно на бесплатных тарифах и с TOTP вместо SMS.
- **С практической:** «всё сразу» — нет; нужна **поэтапная** реализация и тесты.
- **SMS 2FA:** не стоит считать бесплатной частью продукта.

Дальнейший шаг в коде — выбрать **один** бэкенд (рекомендация: **Supabase**) и реализовать **фазу A**, затем **B**. Если решите зафиксировать провайдера — можно добавить в репозиторий скелет клиента Supabase и SQL-миграции.
